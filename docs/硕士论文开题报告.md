# 硕士学位论文开题报告

## 题目：基于大语言模型的 WebAssembly 恶意代码检测与对抗鲁棒性研究

---

## 一、研究背景与问题提出

### 1.1 WebAssembly 的安全威胁

WebAssembly（Wasm）是一种低级字节码格式，设计目标是在 Web 浏览器中以接近原生的速度运行代码。自 2017 年成为 W3C 标准以来，Wasm 已被所有主流浏览器支持，并逐步扩展到服务端（WasmEdge、Wasmtime）、边缘计算（Cloudflare Workers、Fastly Compute@Edge）和嵌入式设备（WAMR）等非浏览器环境。

然而，Wasm 的广泛采用也带来了新的安全威胁：

**（1）加密货币挖矿（Cryptojacking）**

2017 年，CoinHive 事件首次暴露了 Wasm 被用于浏览器端挖矿的威胁。攻击者在网页中嵌入 Wasm 挖矿脚本，利用访客的 CPU 资源挖掘 Monero 等加密货币。据 Palo Alto Networks 2018 年报告，超过 30,000 个网站被发现包含 Wasm 挖矿脚本。Wasm 挖矿脚本相比 JavaScript 实现具有 2-5 倍的性能优势，且更难被传统安全工具检测。

**（2）恶意代码混淆**

Wasm 的二进制格式天然具有混淆特性——相比 JavaScript 源码，Wasm 字节码难以人工阅读和静态分析。Romano 等人在 Wobfuscator（IEEE S&P 2022）中证明，将 JavaScript 恶意代码翻译为 Wasm 可以有效绕过主流杀毒软件的检测。

**（3）供应链攻击**

恶意 Wasm 模块通过 npm 等包管理器传播，嵌入到合法项目中。由于 Wasm 二进制难以审查，这类供应链攻击具有高度隐蔽性。

**（4）跨语言恶意行为**

攻击者利用 JavaScript 和 Wasm 的交互机制，将恶意逻辑分散在两种语言中。Xia 等人在 JWBinder（ESORICS 2023）中发现，主流杀毒软件对 JS-Wasm 多语言恶意代码的检测率仅为 49.1%，远低于对单一语言恶意代码的检测能力。

**（5）向非浏览器环境的扩展趋势**

Wasm 的安全影响正随其应用范围的扩展而持续扩大。除浏览器和服务端外，Wasm 已开始进入嵌入式与资源受限环境。例如，NASA JPL 于 2026 年 7 月开源的 SpaceWASM [21] 将 Wasm 作为嵌入式设备上的指令执行格式，并明确允许"不受信任或低信任的可执行文件"在沙箱中运行。这一趋势表明，Wasm 恶意代码检测的研究价值将持续增长，且检测方法需要具备对不同运行环境的适应能力。

### 1.2 现有检测方法的局限性

当前 Wasm 恶意代码检测方法主要分为三类：

**（1）基于签名的静态检测**

传统杀毒软件依赖已知恶意代码的签名特征进行匹配。然而，Wasm 的混淆特性使得签名匹配方法效果显著下降。JWBinder（ESORICS 2023）在 VirusTotal 上的实测表明，主流杀毒软件对 JS-Wasm 多语言恶意代码的检测率仅为 49.1%。

**（2）基于深度学习的检测**

WasmGuard（WWW 2025）提出了基于 FGSM 对抗训练的深度学习检测方法，在 WasmMal-15K 数据集上达到了 99.20% 的鲁棒准确率和 99.93% 的标准准确率。但该方法存在以下局限：
- 仅做二分类（恶意/良性），无法解释恶意行为类型
- 对新型攻击模式的泛化能力有限
- 对抗训练仅覆盖字节级扰动，未考虑语义级混淆

**（3）基于大语言模型的分析**

StackSight（ICML 2024）和 WaDec（ASE 2024）证明了 LLM 可以理解 Wasm 代码语义并进行反编译。但这些工作聚焦于通用反编译，未针对恶意代码检测场景进行优化。

### 1.3 核心科学问题

基于以上分析，本论文聚焦以下两个核心科学问题：

**问题 1：如何利用大语言模型的代码理解能力，实现高精度、可解释的 Wasm 恶意代码检测？**

现有方法要么依赖浅层特征（签名、统计特征）导致检测率低，要么依赖深度学习黑盒模型导致不可解释。LLM 具有强大的代码语义理解能力，但如何设计有效的特征提取和提示工程策略，将 Wasm 的结构化特征有效传递给 LLM，是一个尚未解决的问题。

**问题 2：Wasm 恶意代码检测方法在面对对抗性混淆攻击时是否可靠？如何增强其鲁棒性？**

攻击者可以使用多种混淆手段（函数名重命名、指令替换、死代码注入、数据段加密、控制流平坦化）来绕过检测。现有研究尚未系统评估 Wasm 恶意检测方法在各种混淆攻击下的鲁棒性，也未提出针对性的防御增强方法。

## 二、研究内容与方法

### 2.1 创新点 1：基于多维度特征提取与两阶段 LLM 分析的 Wasm 恶意代码检测方法

#### 2.1.1 方法概述

提出一种"预设关键词初筛 + LLM 多维度综合分析 + LLM 代码验证"的混合二阶段检测方法（V3），核心思想是充分利用 LLM 的代码理解能力，同时通过领域知识注入解决 LLM 的"过度合理化"问题。

#### 2.1.2 技术路线

```
阶段 0：多维度特征提取（确定性，无需 LLM）
  │
  ├── 结构层：imports/exports 函数签名、数量
  ├── 数据层：数据段字符串、可疑字符串
  ├── 行为层：指令频率统计、循环密度、嵌套深度
  └── 高级特征：代码熵、框架识别、混淆检测、挖矿上下文模式、可疑度评分
  │
  ▼
阶段 1：LLM 多维度综合分析（第 1 次 LLM 调用）
  │
  ├── 函数筛选：验证预设标记 + 发现遗漏的可疑函数
  ├── 数据层分析：字符串与可疑函数的关联性
  ├── 行为层分析：指令模式是否支持恶意判断
  └── 综合判定：基于所有维度给出初步判定
  │
  ▼
阶段 1.5：可疑函数代码提取（确定性，无需 LLM）
  │
  ├── 从 WAT 中定位可疑函数的函数体
  └── 截取代码片段（每个函数最多 200 行，最多 3-5 个函数）
  │
  ▼
阶段 2：LLM 代码验证（第 2 次 LLM 调用）
  │
  ├── 用代码证据验证/修正阶段 1 的初步判定
  └── 输出最终判定 + 可解释性报告
```

#### 2.1.3 关键技术创新

**（1）多维度语义特征提取**

不同于简单截取 WAT 文本，本方法从四个层次提取结构化特征：
- 结构层：完整的导入/导出函数列表，而非截断的代码片段
- 数据层：数据段中的字符串信息，可暴露恶意库路径（如 `cryptonight.c`）
- 行为层：指令频率统计，识别计算密集型代码模式
- 高级特征：代码熵、框架识别、混淆检测、挖矿上下文模式分析

**（2）LLM 智能函数筛选**

在阶段 1 中，LLM 不仅验证预设关键词标记的函数是否真的可疑（排除误报），还从剩余函数中发现预设关键词遗漏的可疑函数。这解决了纯关键词匹配的局限性。

**（3）领域知识注入**

在 Prompt 中注入 Wasm 安全领域的专业知识，解决 LLM 的"过度合理化"问题。例如：
- CryptoNight/hash_cn 在 Web Wasm 中没有合法用途
- CryptoJacking 的 JS+Wasm 分离架构特征
- 合法哈希与恶意哈希的区分标准

#### 2.1.4 实验验证

在 WasmMal-15K 数据集（WWW 2025 发布，15,024 个样本）上进行验证。初步实验结果（30 个样本，15 良性 + 15 恶意）：

| 指标 | V1（WAT 截断） | V2（多维度特征） | V3（两阶段+领域知识） |
|------|---------------|-----------------|---------------------|
| 准确率 | 72.4% | 90.0% | **100%** |
| 精确率 | 100% | 100% | **100%** |
| 召回率 | 42.9% | 80.0% | **100%** |
| F1 | 60.0% | 88.9% | **100%** |

### 2.2 创新点 2：Wasm 恶意代码检测的对抗鲁棒性评估与增强

#### 2.2.1 问题动机

现有 Wasm 恶意检测方法（包括本论文的 V3 方法）在面对攻击者的主动对抗时是否可靠？攻击者可以使用多种混淆手段来绕过检测：

- **函数名重命名**：将 `_cryptonight_hash` 重命名为 `_calculate` 或随机字符串
- **指令替换**：用等价但不同的指令序列替换原始指令
- **死代码注入**：插入大量无意义代码稀释特征
- **数据段加密**：加密数据段中的恶意字符串（如 `cryptonight.c`）
- **控制流平坦化**：将结构化控制流转换为平坦的 switch-case 结构

此外，随着 Wasm 向嵌入式等非浏览器环境扩展（如 NASA JPL 的 SpaceWASM [21] 已允许不受信任的 Wasm 模块在沙箱中执行），对抗攻击的现实威胁将进一步增大。然而，目前 Wasm 恶意检测的对抗鲁棒性研究**完全空白**（DBLP 检索 0 篇），这是一个亟待填补的研究空白。

#### 2.2.2 技术路线

**（1）攻击方法实现**

参考现有 Wasm 混淆研究，实现 5 种对抗攻击方法：

| 攻击方法 | 参考论文 | 攻击目标 |
|---------|---------|---------|
| 函数名重命名 | WASMixer (ESORICS 2024) | 绕过函数名关键词检测 |
| 指令替换 | WeBMO (COMPSAC 2024) | 改变指令频率特征 |
| 死代码注入 | WASMixer (ESORICS 2024) | 稀释统计特征 |
| 数据段加密 | Cryptic Bytes (arXiv 2024) | 隐藏数据段恶意字符串 |
| 控制流平坦化 | WASMixer (ESORICS 2024) | 改变控制流特征 |

**（2）鲁棒性评估框架**

设计系统化的评估框架：
- 对 WasmMal-15K 中的恶意样本分别施加 5 种攻击
- 评估 V3 方法在各种攻击下的检测率下降
- 绘制"攻击强度 vs 检测率"曲线
- 分析每种攻击对不同检测阶段的影响

**（3）防御增强方法**

提出针对性的防御增强方法：
- **对抗训练**：使用混淆样本增强训练数据
- **特征鲁棒化**：设计对混淆不敏感的特征（如函数调用图结构特征）
- **领域知识增强**：在 Prompt 中增加对混淆模式的识别指导
- **集成检测**：结合多种检测策略降低单一方法的脆弱性

#### 2.2.3 预期贡献

- 首次系统评估 Wasm 恶意检测方法的对抗鲁棒性
- 提供 5 种可复现的 Wasm 混淆攻击实现
- 提出有效的防御增强方法，提升检测方法的实际可靠性

## 六、备选方向（工作量不足时的补充）

### 方向 B：基于 Wasm 沙箱的 LLM Agent 安全执行框架

**题目**：面向大模型 Agent 的 WebAssembly 安全沙箱执行环境

**背景**：随着 LLM Agent（如 OpenAI Code Interpreter、Devin、Cursor）的普及，Agent 自主生成并执行代码的场景日益增多。然而，Agent 生成的代码可能包含恶意行为（如数据窃取、系统破坏），需要安全的执行环境。

**核心思路**：利用 Wasm 的沙箱隔离特性，为 LLM Agent 生成的代码提供安全的执行环境。结合 V3 检测方法，实现"静态检测 + 运行时隔离"的双重防护。

**创新点**：
1. 面向 LLM Agent 的 Wasm 沙箱权限模型设计（基于 WASI capability-based 权限）
2. 结合静态检测（V3）和运行时监控的混合安全防护

**可行性**：
- Wasm 沙箱是 Wasm 的核心设计特性，技术成熟
- WASI 0.2/0.3 提供了标准化的权限控制接口
- Cloudflare Workers 等产品已验证了 Wasm 沙箱的实际可用性
- 与 V3 检测方法形成"检测 + 防护"的完整安全方案
