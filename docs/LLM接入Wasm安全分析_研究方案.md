# LLM 接入 Wasm 安全分析：研究方案

> 核心问题：Wasm 二进制怎么喂给 LLM？LLM 怎么做安全分析？

---

## 一、核心挑战：Wasm 二进制 → LLM 输入

LLM 接受的是**文本**，而 Wasm 是**二进制字节流**。第一步必须做**表示转换**。

### 三种可行的输入表示

| 表示方式 | 做法 | 优点 | 缺点 |
|---------|------|------|------|
| **WAT 文本格式** | 用 `wasm2wat` 将二进制转为文本 | 保留完整语义，LLM 可直接阅读 | 文件较大，可能超出上下文窗口 |
| **反汇编指令序列** | 提取函数级指令序列 | 粒度可控，可按函数切分 | 丢失部分结构信息 |
| **结构化摘要** | 提取 imports/exports/内存布局/函数签名等元信息 | 体积小，信息密度高 | 丢失具体实现细节 |

**推荐组合**：结构化摘要（全局） + 关键函数的 WAT 文本（局部）

---

## 二、三种 LLM 接入架构

### 架构 A：LLM 直接分类（最简单）

```
.wasm 文件
    ↓
wasm2wat 转文本
    ↓
截取/分块（处理长文本）
    ↓
Prompt: "分析以下 WebAssembly 代码，判断是否为恶意代码，并说明理由"
    ↓
LLM 输出：{分类结果, 恶意类型, 解释}
```

**优点**：实现简单，可解释性天然具备
**缺点**：LLM 推理慢、成本高，大规模检测不现实
**适合**：作为 WasmGuard 的"第二层"——WasmGuard 快速筛选可疑样本，LLM 深度分析

### 架构 B：LLM 辅助特征工程 + 轻量分类器（推荐）

```
.wasm 文件
    ↓
┌───────────────────────────────────┐
│  传统特征提取（字节统计、段分析）    │  ← 快速、低成本
│  LLM 语义特征提取（函数意图分析）    │  ← 深度、可解释
└───────────────────────────────────┘
    ↓
特征融合
    ↓
轻量分类器（XGBoost/小型NN）
    ↓
输出：{分类, 恶意类型, 解释}
```

**优点**：兼顾效率和深度，LLM 只在特征提取阶段使用
**缺点**：需要设计特征融合策略
**适合**：硕士论文，两个创新点清晰

### 架构 C：两阶段级联（最实用）

```
.wasm 文件
    ↓
第一阶段：WasmGuard 快速检测（毫秒级）
    ↓
判定为可疑？ ──否──→ 放行
    ↓ 是
第二阶段：LLM 深度分析
    ↓
输出：
  - 恶意行为分类（挖矿/混淆/窃取/...）
  - 自然语言解释（"该模块包含..."）
  - 风险评估（高/中/低）
  - 建议处置措施
```

**优点**：效率最高，LLM 只分析可疑样本
**缺点**：依赖第一阶段检测质量
**适合**：工程落地，论文故事也好讲

## 三、推荐的研究方案

### 方案一：基于架构 B（推荐作为硕士论文）

**创新点 1（Wasm 侧）**：Wasm 多粒度语义特征提取
- 用 WABT 工具提取 Wasm 的结构化元信息（imports、exports、段布局）
- 用 `wasm2wat` 提取关键函数的指令序列
- 将两者融合为结构化的"语义描述"

**创新点 2（LLM 侧）**：面向 Wasm 安全分析的 LLM Prompt 工程 + 微调
- 设计 Wasm 安全领域的专用 Prompt 模板
- 在 WasmMal-15K 上对开源 LLM（如 LLaMA/Qwen）做 LoRA 微调
- 让 LLM 输出结构化结果：恶意类型 + 置信度 + 自然语言解释

**实验设计**：
1. 在 WasmMal-15K 上对比：纯 LLM vs WasmGuard vs 你的方法
2. 评估维度：准确率、恶意分类精度、可解释性（人工评估）、推理速度
3. 零样本实验：构造训练集中没有的新型恶意样本，测试泛化能力

### 方案二：基于架构 C（更偏工程）

**创新点 1**：WasmGuard + LLM 级联检测框架
**创新点 2**：LLM 驱动的 Wasm 恶意行为语义分析与报告生成

## 五、最小可行实验（MVP）

### 目标

验证 LLM 是否真的能理解 Wasm 代码并给出有意义的安全分析。一天之内可跑通。

### 前置准备

1. 安装 WABT 工具：`brew install wabt`
2. 准备几个 .wasm 样本（良性 + 恶意各几个）
3. 准备 LLM 访问（API 或本地部署）

### 实验代码

```python
import subprocess
import json

# 1. 将 Wasm 转为 WAT 文本
def wasm_to_wat(wasm_path):
    result = subprocess.run(
        ["wasm2wat", wasm_path],
        capture_output=True, text=True
    )
    return result.stdout

# 2. 提取结构化摘要
def extract_summary(wasm_path):
    result = subprocess.run(
        ["wasm-objdump", "-x", wasm_path],
        capture_output=True, text=True
    )
    return result.stdout

# 3. 构造 Prompt
def build_prompt(wat_snippet, summary):
    return f"""你是一个 WebAssembly 安全分析专家。
请分析以下 WebAssembly 模块，判断其是否为恶意代码。

## 模块摘要信息
{summary}

## 关键函数代码（WAT 格式）
{wat_snippet[:3000]}

请按以下格式输出分析结果：
1. 判定结果：恶意 / 良性
2. 恶意类型（如为恶意）：挖矿 / 混淆 / 数据窃取 / 广告欺诈 / 其他
3. 置信度：高 / 中 / 低
4. 判断依据：（详细说明你从代码中观察到的特征）
5. 行为描述：（用自然语言描述该模块的实际行为）
"""

# 4. 调用 LLM（以 OpenAI API 为例，也可替换为本地模型）
def analyze_wasm(wasm_path, llm_client):
    wat = wasm_to_wat(wasm_path)
    summary = extract_summary(wasm_path)
    prompt = build_prompt(wat, summary)
    
    response = llm_client.chat.completions.create(
        model="qwen2.5-14b-instruct",  # 或其他模型
        messages=[{"role": "user", "content": prompt}],
        temperature=0.1
    )
    return response.choices[0].message.content

# 5. 批量测试
def batch_test(wasm_dir, llm_client):
    results = []
    for f in os.listdir(wasm_dir):
        if f.endswith(".wasm"):
            analysis = analyze_wasm(os.path.join(wasm_dir, f), llm_client)
            results.append({"file": f, "analysis": analysis})
    return results
```

### 验证步骤

1. 选 10 个已知良性 .wasm + 10 个已知恶意 .wasm
2. 用上述代码逐个分析
3. 记录 LLM 的判定结果和分析内容
4. 计算准确率，人工评估解释质量
5. 如果准确率 > 70% 且解释合理 → 方向可行，继续深入

## 七、创新点 1 的小论文可行性分析

### 7.1 现有内容已具备的论文要素

创新点 1（两阶段 LLM 检测方法）单独已具备发表 CCF-C 小论文的条件，现有要素：

- **新方法**：两阶段 LLM 检测 + 多维度特征提取，有明确技术贡献
- **可解释性输出**：LLM 生成自然语言判定理由，是 DL 方法（WasmGuard）做不到的核心差异化
- **数据集增强**：为 WasmMal-15K 每条样本生成自然语言标注，副产品有独立价值

论文故事线：

> "现有方法是黑盒的，我们用 LLM 实现了可解释的 Wasm 恶意检测，同时构建了首个带自然语言标注的 Wasm 恶意代码数据集。"

目标投稿方向：ISSTA、DBSec、ICICS（CCF-C），或 ESORICS、AsiaCCS（CCF-B）。

### 7.2 还差一块：对比基线不够有力

审稿人必问："和 WasmGuard（99.2% 鲁棒准确率）比，优势在哪？"

回答是"可解释性"，但审稿人会追问：**可解释性有没有被量化评估？**

需要补充的三个实验：

| 实验 | 目的 |
|------|------|
| 在完整 WasmMal-15K 上跑 V3 | 证明大规模下仍有竞争力的准确率 |
| 对比 WasmGuard 的准确率 | 明确定位：不是"更准"而是"更可解释" |
| 人工评估自然语言解释质量 | 量化可解释性，20-30 个样本的人工评分即可 |

### 7.3 数据集标注贡献的包装方式

"为 WasmMal-15K 生成自然语言标注"这个贡献需要单独强调，建议在论文中这样表述：

> 我们构建了 **WasmMal-Explain**，首个附带自然语言恶意行为描述的 Wasm 数据集，可用于 LLM 微调、检测方法评估和安全人员培训。

数据集贡献在安全领域论文审稿中是实质性加分项。

### 7.4 核心定位建议

**不要**在准确率上和 WasmGuard 硬比，而是主打以下两点差异化：

1. **可解释性**：WasmGuard 只输出 0/1，V3 输出自然语言报告，这在实际安全运营中有真实价值
2. **数据集贡献**：WasmMal-Explain 是社区可复用的资源，审稿人认可这类工作

### 7.5 一句话结论

创新点 1 单独够一篇 CCF-C。需要补的只有一件事：在完整数据集上跑出实验结果，加一个人工评估解释质量的小实验。

## 八、真实样本分析实验记录

### 8.1 实验设置

对两个真实 wasm 样本分别进行静态分析：
- 恶意样本：`00a02b35...wasm`（来自 WasmMal 数据集）
- 正常样本：`0a0eedb4...wasm`（Rust wasm-bindgen 编译的日程管理库）

### 8.2 调用图对比结果

| 指标 | 恶意样本 | 正常样本 |
|------|---------|---------|
| 函数数量 | 73 | 239 |
| 调用关系数 | 66 | 640 |
| 图密度 | 0.012 | 0.011 |
| 函数名 | 全部混淆为数字 | 全部混淆为数字 |

**结论：调用图对这两个样本完全没有区分度。** 两者图密度几乎相同，无法判断恶意与否。

### 8.3 调用图失效的根本原因

恶意样本是一个**库型 wasm**，其核心函数 `_cryptonight_hash` 通过导出表暴露给 JavaScript 调用，真正的高频调用发生在 wasm 模块外部（宿主 JS），wasm 内部调用图看不到这条调用链。

> **调用图对库型 wasm 恶意代码基本失效，仅对独立可执行型 wasm 有一定判别力。**

### 8.4 恶意样本的真实性验证

通过深入分析 `_cryptonight_hash`（func 70，约 1400 行 WAT）的指令内容，确认这是真正的 CryptoNight 挖矿算法实现，不是仅仅起了个名字：

**证据 1：2MB Scratchpad 魔法常数**

```wasm
i32.const 2097152   ← 2MB = 2 × 1024 × 1024，CryptoNight 核心内存大小
i32.const 2097536   ← 2MB + 384，CryptoNight 状态结构偏移
```

主循环以 128 字节为步长遍历整个 2MB scratchpad，这是 CryptoNight 算法的核心机制，该常数在其他场景下没有出现理由。

**证据 2：循环内连续 8 次 AES 轮调用**

```wasm
loop
  call 18  ; AES round × 8（连续调用）
  ...（128 字节写回 scratchpad）
  i32.const 2097152
  i32.lt_u
  br_if 0
end
```

每次循环迭代调用 8 轮 AES 并写入 128 字节，这是 CryptoNight 内存初始化阶段的标准实现。

**证据 3：数据段中的断言字符串**

```
lib/cryptonight/cryptonight.c
do_jh_hash / (SUCCESS == r)
do_skein_hash / (SKEIN_SUCCESS == r)
```

这是 C 源码 `assert()` 宏的残留。JH 和 Skein 哈希是 CryptoNight 最终阶段的组成部分，进一步印证算法真实性。

### 8.5 关于误检问题的分析

**问题**：如果攻击者只是把函数起名为 `_cryptonight_hash` 而不实现真正的挖矿逻辑，V3 方法会误检吗？

**结论：误检概率极低。** V3 是多维度交叉验证，不依赖单一信号：

| 检查项 | 只复制名字 | 真实恶意样本 |
|--------|---------|------------|
| 函数名 `_cryptonight_hash` | ✓ | ✓ |
| 数据段含 `cryptonight.c` 字符串 | ✗ | ✓ |
| 常数 `i32.const 2097152`（2MB） | ✗ | ✓ |
| 循环内重复 AES 调用模式 | ✗ | ✓ |
| 高计算密集型指令熵 | ✗ | ✓ |

LLM 在第二阶段读到函数体后会判断"名字可疑但实现不符"，从而避免误检。

### 8.6 混淆后的漏检风险与混淆不变特征

**核心问题**：如果攻击者对恶意样本做彻底的语义混淆（函数名重命名 + 数据段加密 + 导出名混淆），V3 方法会大量漏检。

但 CryptoNight 这类算法有一类**混淆无法消除的特征**——改了就不能运行：

| 特征 | 能被混淆吗 | 检测方法 |
|------|-----------|---------|
| 函数名、导出名、字符串 | 能 | LLM 语义分析 |
| 魔法常数 `2097152`（2MB） | **不能** | 脚本扫描常数 |
| AES S-Box 字节分布（4KB 均匀熵） | **不能** | 数据段熵分析 |
| 单函数局部变量数（47 个） | 难 | 静态复杂度统计 |
| 调用图拓扑 | 对库型无效 | 仅适用独立程序 |

### 8.7 对创新点 2 的研究意义

本实验揭示了一个比"调用图有效"更有价值的发现：

> **对库型 wasm 恶意代码，函数调用图完全失效；真正的混淆不变特征是算法内嵌的魔法常数和数据段的统计指纹（熵分布）。**

这个结论可以成为创新点 2 的核心 insight，重新定位论文叙事：

- 不是"我们用调用图增强了防御"
- 而是"**我们系统分析了各类特征在混淆攻击下的存活性，发现算法魔法常数和数据段熵是混淆不变特征，可作为 LLM 语义信号失效后的补偿**"

这是一个诚实且有学术深度的结论，比单纯宣称"我们达到了 XX% 准确率"对审稿人更有说服力。
