# V3 混合二阶段分析方案

## 核心思路

**充分利用 LLM 的理解能力**，采用"预设关键词初筛 + LLM 多维度综合分析 + LLM 代码验证"的混合二阶段方案。

**关键设计原则**：
- 阶段 0 已经提取了所有维度的特征（结构/数据/行为/高级），不应该只让 LLM 做函数名筛选
- 阶段 1 应该把**所有维度的特征**一次性喂给 LLM，让它做多维度综合分析
- 阶段 2 聚焦于**代码层面的验证**，确认或修正阶段 1 的初步判定

## 完整流程

```
输入: .wasm 文件
  │
  ▼

│ 阶段 0：特征提取（确定性，无需 LLM）            │

│  - 高级特征：熵 / 框架识别 / 混淆检测 /        │

## 各阶段详细说明

### 阶段 0：特征提取（确定性，无需 LLM）

复用 V2 的特征提取器，提取以下特征：

**结构层**：
- 所有导入函数（imports）
- 所有导出函数（exports）
- 导入/导出数量

**数据层**：
- 数据段字符串
- 可疑字符串（基于预设关键词）

**行为层**：
- 指令频率统计（算术/位运算/内存/控制流/比较）
- 函数数量、循环数量、循环密度
- 最大嵌套深度

**高级特征**：
- 代码熵（整体二进制熵）
- 预设关键词匹配的初步可疑导出函数
- 混淆函数名检测
- 挖矿上下文模式
- 框架识别（EOSIO/Emscripten/Rust）
- 可疑度评分

**输出**：
- `features`: 完整的特征字典

### 阶段 1：LLM 多维度综合分析

**目的**：充分利用 LLM 的理解能力，一次性对所有维度的特征进行综合分析。

**与旧方案的区别**：
- 旧方案：阶段 1 只做函数名筛选，浪费 LLM 能力
- 新方案：阶段 1 同时分析结构/数据/行为/高级特征，输出多维度结论 + 初步判定

**输入给 LLM**：
```
你是一个 WebAssembly 安全分析专家。请对以下 WebAssembly 模块进行多维度综合分析。

## 模块概览
- 文件大小: {file_size} bytes
- 导出函数数: {export_count}
- 导入函数数: {import_count}
- 代码熵: {code_entropy} (>6.5 可能表示加密/压缩)
- 识别框架: {framework}
- 可疑度评分: {suspicion_score}/100

## 结构层 - 导出函数
所有导出: {all_exports}
预设关键词标记的可疑函数: {preset_suspicious_exports}
混淆函数名: {obfuscated_exports}

## 结构层 - 导入函数
所有导入: {all_imports}

## 数据层
数据段字符串数: {data_string_count}
字符串样本（前 10 个）: {data_strings_sample}
可疑字符串: {suspicious_strings}

## 行为层
- 函数数量: {function_count}
- 循环数量: {loop_count}
- 循环密度: {loop_density} (每千行)
- 最大嵌套深度: {max_nesting_depth}
- 指令统计:
  - 算术运算: {arithmetic}
  - 位运算: {bitwise} ⚠ 高频（如 >1000）
  - 内存操作: {memory}
  - 控制流: {control}
  - 比较运算: {comparison}

## 高级特征
- 挖矿上下文模式: {mining_context}

请完成以下多维度分析：

1. **函数筛选**：
   - 验证预设标记的函数是否真的可疑（排除误报，如区块链合约的合法哈希函数）
   - 从剩余函数中发现可能遗漏的可疑函数

2. **数据层分析**：
   - 数据段字符串是否与可疑函数关联？
   - 是否暴露恶意意图（如挖矿库路径、加密算法名称）？

3. **行为层分析**：
   - 指令统计模式是否支持恶意判断？
   - 高位运算 + 哈希导出是否指向挖矿算法？

4. **综合判定**：
   - 基于所有维度的证据，给出初步恶意/良性判定

输出 JSON 格式：
{
  "confirmed_suspicious": ["确认可疑的函数名"],
  "new_suspicious": ["新发现的可疑函数名"],
  "false_positives": ["排除的误报函数名"],
  "data_analysis": "数据层分析结论",
  "behavior_analysis": "行为层分析结论",
  "preliminary_verdict": "malicious" 或 "benign",
  "preliminary_type": "cryptojacking" / "obfuscation" / "data_exfiltration" / "ad_fraud" / "other" / "none",
  "preliminary_confidence": "high" / "medium" / "low",
  "reasoning": "综合分析依据"
}
```

**输出**：
- `final_suspicious = confirmed_suspicious + new_suspicious`（用于阶段 1.5 代码提取）
- `preliminary_verdict`（初步判定，传给阶段 2 验证）
- `data_analysis` / `behavior_analysis`（多维度分析结论，传给阶段 2 参考）

**Token 消耗**：中（所有维度特征 + Prompt，约 1000-3000 tokens）

### 阶段 1.5：代码提取（确定性，无需 LLM）

**目的**：从 WAT 中提取最终可疑函数的实际代码。

**步骤**：
1. 解析 WAT 中的 export → func index 映射
   - 格式：`(export "name" (func N))`
2. 根据 func index 定位函数体
   - 格式：`(func (;N;) ...)`
3. 截取函数体代码（每个函数最多 200 行）
4. 限制最多提取 3-5 个函数的代码

**输出**：
- `suspicious_func_code`: {函数名: 函数体代码}

### 阶段 2：LLM 代码验证

**目的**：用代码证据验证或修正阶段 1 的初步判定。

**与旧方案的区别**：
- 旧方案：阶段 2 从零开始分析，与阶段 1 有重复劳动
- 新方案：阶段 2 聚焦于代码验证，输入阶段 1 的结论，用代码确认/修正

**输入给 LLM**：
```
你是一个 WebAssembly 安全分析专家。

在上一轮分析中，我们基于模块的多维度特征给出了初步判定。现在请根据可疑函数的实际代码，验证或修正这个判定。

## 上一轮分析结论
- 初步判定: {preliminary_verdict}
- 初步类型: {preliminary_type}
- 初步置信度: {preliminary_confidence}
- 分析依据: {reasoning}
- 数据层分析: {data_analysis}
- 行为层分析: {behavior_analysis}

## 可疑函数代码
{func_name_1}:
```wat
{code_1}
```

{func_name_2}:
```wat
{code_2}
```

**代码分析指导**：
- 验证上一轮判定是否被代码证据支持
- 如果代码证据与初步判定矛盾，请修正判定
- 区分合法哈希计算（SHA-256 校验、区块链验证）和恶意挖矿（CryptoNight、RandomX）
- 关注代码中的特征模式：
  - 挖矿：大量位运算、循环、特定常量（0x6a09e667）、PoW 计算逻辑
  - 合法哈希：标准 API 调用、明确的业务逻辑、框架特征

请按以下 JSON 格式输出最终分析结果：
{
  "verdict": "malicious" 或 "benign",
  "verdict_change": true 或 false（是否修正了初步判定）,
  "malware_type": "cryptojacking" / "obfuscation" / "data_exfiltration" / "ad_fraud" / "other" / "none",
  "confidence": "high" / "medium" / "low",
  "code_evidence": "代码层面的关键证据",
  "reasoning": "最终判断依据，结合特征分析和代码证据",
  "behavior_description": "用自然语言描述该模块的实际行为"
}
```

**输出**：
- `verdict`: 最终判定
- `verdict_change`: 是否修正了初步判定
- `malware_type`: 恶意类型
- `confidence`: 置信度
- `code_evidence`: 代码层面的关键证据
- `reasoning`: 最终判断依据
- `behavior_description`: 行为描述

**Token 消耗**：中（阶段 1 结论 + 代码片段 + Prompt，约 3000-8000 tokens）

## 与 V2 的对比

| 维度 | V2 | V3 |
|------|-----|-----|
| **可疑函数识别** | 纯预设关键词 | 预设关键词 + LLM 智能筛选 |
| **多维度分析** | 特征提取后直接喂给 LLM 做一次性判断 | 阶段 1 做多维度综合分析 + 初步判定 |
| **代码分析** | 无 | 阶段 2 提取可疑函数代码做验证 |
| **LLM 调用次数** | 1 次 | 2 次 |
| **阶段 1 的角色** | 全部分析 | 多维度综合分析 + 初步判定 |
| **阶段 2 的角色** | 不存在 | 代码验证（确认/修正初步判定） |
| **误报处理** | 无法排除预设关键词的误报 | LLM 在阶段 1 排除误报 |
| **漏报处理** | 无法发现预设关键词之外的模式 | LLM 在阶段 1 发现新的可疑模式 |
| **分析深度** | 基于特征的启发式判断 | 特征综合分析 + 代码验证 |
| **可解释性** | 引用特征数据 | 多维度分析 + 代码证据链 |
| **判定一致性** | 单次判定 | 两阶段交叉验证，更可靠 |

## 每样本 Token 消耗估算

| 调用 | 输入内容 | 估算 Token |
|------|---------|-----------|
| 第 1 次（多维度综合分析） | 所有维度特征 + Prompt | ~1000-3000 |
| 第 2 次（代码验证） | 阶段 1 结论 + 代码片段 + Prompt | ~3000-8000 |
| **合计** | | **~4000-11000** |

## 预期效果

### 减少假阳性（V2 的 EOSIO 案例）
- V2：预设关键词命中 `getTransactionHash`，可疑度 100/100，判定恶意
- V3 阶段 1：LLM 综合分析发现 EOSIO 框架特征 + 合法区块链 API，初步判定良性
- V3 阶段 2：代码验证确认是合法的哈希校验逻辑，最终判定良性

### 减少假阴性（V2 的混淆挖矿案例）
- V2：预设关键词未命中某些混淆函数，漏检
- V3 阶段 1：LLM 从剩余函数中发现可疑的混淆命名模式 + 高位运算 + 挖矿上下文，初步判定恶意
- V3 阶段 2：代码验证识别出 CryptoNight 算法特征，最终判定恶意

### 提升可解释性
- V2 推理："位运算 3254 次，高度可疑"
- V3 推理：
  - 阶段 1："函数名混淆 + 位运算 3254 次 + 挖矿上下文模式 + 数据段含 cryptonight 路径 → 初步判定恶意"
  - 阶段 2："函数 `_wbYnAgshchW_hash` 的代码包含典型的 CryptoNight 计算模式：大量 XOR/ROTL 操作、特定常量（0x6a09e667）、嵌套循环结构 → 确认恶意"

### 两阶段交叉验证
- 阶段 1 和阶段 2 的判定一致 → 高置信度
- 阶段 1 和阶段 2 的判定不一致 → 需要人工审查，或标记为低置信度

## 潜在风险与缓解

| 风险 | 缓解方案 |
|------|---------|
| 成本翻倍（2 次 LLM 调用） | 第 1 次调用 token 消耗适中，总成本增加约 50-80% |
| 延迟增加 | 两次调用串行，总延迟约增加 100% |
| LLM 筛选不一致性 | 使用 temperature=0.1，减少随机性 |
| WAT 函数体解析复杂 | 简化正则 + 合理截断（max_lines=200） |
| 函数体过长 | 限制每个函数最多 200 行，最多 3-5 个函数 |
| 阶段 1 和阶段 2 判定冲突 | 以阶段 2（代码验证）为准，标记为低置信度 |
